# gmsynth — 64-voice GM synthesizer for Sony Spresense

A General MIDI software synthesizer for the Sony Spresense board
(CXD5602), rendering 64 voices at 48 kHz across the multicore ASMP
architecture:

- **Main core**: SMF (Standard MIDI File) streaming sequencer, global
  64-voice ledger with voice stealing, per-song sample loading from a
  compact ADPCM sound bank.
- **4 ASMP subcores** (`worker/gmvoice`): identical renderer workers,
  16 voices each, commanded per audio block by the supervisor.

The engine reads its instrument data at runtime from a single binary
sound bank file, `/mnt/spif/gmbank.bin`, which **you build yourself
from any GM-compatible SoundFont (.sf2)** — see below.

## Building the sound bank

The sound bank is **not** included in this repository (see Licensing).
Build it from a GM SoundFont of your choice:

```sh
# Optional: scan your own MIDI collection to allocate the size budget
python3 corpus_scan.py /path/to/midis --json corpus_stats.json

python3 gmbank_build.py /path/to/your_font.sf2 \
    [--stats corpus_stats.json] [--out gmbank.bin]

python3 bank_qa.py gmbank.bin          # offline sanity check
python3 gmrender.py gmbank.bin song.mid --out out.wav   # offline listen test
```

Size is capped automatically: `--budget 3.5M` (on `gmbank_build.py` or
`bank_calibrate.py`) runs a closed loop that scales the global quality
knob (sample rates, loop lengths, zone density) until the bank fits,
so even multi-GB fonts land on the SPI flash. At runtime the device
fits each song into the 512KB shared sample pool by thinning
many-zone programs and, as a last resort, dropping the least-played
one; `gmrender.py --pool` models that exact logic offline so what you
verify is what the board plays.

Or let `bank_calibrate.py` do the whole loop — build the bank, then
render a self-generated calibration MIDI (one note per multisample
zone in the bank's own region table, plus every drum key) through both
fluidsynth (ground truth) and the offline device model, compare
levels, and patch corrective factors straight into each region's gain
field until every zone sits within a tolerance (default ±1 dB) of the
source font's own balance:

```sh
python3 bank_calibrate.py /path/to/your_font.sf2 \
    --out gmbank.bin --gains-out zone_gains.json --report calib_report.txt
```

Zone-level granularity means a register that came out hot is fixed
independently of the rest of its program, and a tier-thinned zone set
is calibrated exactly as built. Iterations only re-render and patch
the `.bin` (no rebuild), so they take seconds. `calib_report.txt` is
the equivalence proof (before/after residual per zone);
`zone_gains.json` is the applied recipe; requires `fluidsynth` on
PATH. No MIDI files are needed — the calibration score is generated by
the tool. Works with any font, so the balance always matches *your*
chosen reference.

Calibration *converges* a bank; to *judge* one without touching it,
use check mode — one measurement, no gain patching, exit code 1 if any
audible zone is outside tolerance:

```sh
python3 bank_calibrate.py /path/to/your_font.sf2 --bank gmbank.bin \
    --check --tol 1.0 --resid-out calib_residuals.json
```

Because the calibration score is generated from the bank's own region
table, a region that a builder regression silently drops also
disappears from the test. Pin the structure separately:

```sh
python3 bank_manifest.py gmbank.bin --write   # once, from a good bank
python3 bank_manifest.py gmbank.bin           # exit 1 on any drift
```

Then copy `gmbank.bin` to the board's SPI flash (`/mnt/spif/gmbank.bin`)
along with the `gmvoice` worker ELF (`/mnt/spif/gmvoice`).

Any GM-compatible SF2 works: FluidR3 GM (MIT), GeneralUser GS (its own
permissive license), SGM-V2.01 (GPLv3), TimGM6mb (GPLv2), etc. The
builder handles multisample zones, key/velocity ranges, loop points,
tuning generators, and encodes everything to ADPCM within a size budget.

## Regression tests

The C sequencer/ledger logic is regression-tested **on the host**: the
real `gmsynth_main.c` / `gm_seq.c` / `gm_bank.c` are compiled against
stub NuttX/ASMP headers (`tests/host/stubs/`), driven over fixed MIDI
cases (CC/bend vs note ordering, sustain, self/global steal, attack
guard, drum choke, CC111 loop, event overflow, tempo change, running
status, multi-track merge, RPN bend range), and the per-block worker
event streams are compared against committed JSON expectations:

```sh
sh tests/smoke.sh                     # CLI/README contract smoke test
python3 tests/trace_regression.py     # build + run all trace cases
python3 tests/trace_regression.py --update   # bless new expectations
```

Everything runs against a bank built from the repo-owned deterministic
fixture font (`tests/make_fixture_sf2.py`) — no third-party SoundFont,
MIDI files or hardware needed — and the fixture bank's structure is
pinned by `tests/expected/fixture_manifest.json`. Requires only `gcc`
and `numpy`; CI runs the same two commands.

The offline renderer is a *behavioral* model of the device engine, not
a bit-accurate one: the device advances sample positions in 16.16
fixed-point and runs a sequential one-pole echo LPF, while the model
uses floats and a truncated-FIR approximation. Hardware-in-the-loop
comparison (device PCM capture vs the model) is future work; see
`docs/validation.md` for what has and has not been verified on the
board.

## Building the firmware

This directory is a Spresense SDK application. Place (or symlink) it
into your `spresense` tree, e.g. as `synthapps/gmsynth`, make sure the
containing directory is registered as an application directory, enable
`CONFIG_SYNTHAPPS_GMSYNTH`, then build the SDK as usual. At boot the
ROMFS init script starts `gmsynth`, which plays SMF files from
`/mnt/sd0` or `/mnt/spif`.
